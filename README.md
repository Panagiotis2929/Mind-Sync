## What Is This?

Mind-Sync is a **real-time generative audio synthesis engine** that uses **pure mathematics** to produce brainwave entrainment audio вҖ” no pre-recorded MP3s, no third-party audio APIs, no cloud dependencies.

The backend **computes mathematical audio blueprints** (frequency specifications, amplitude envelopes, noise spectral parameters) via a custom DSP engine. The frontend **synthesizes the actual audio in real-time** using the Web Audio API, driven entirely by the blueprint's mathematical parameters.

**This is not a music player. It is an audio computation engine.**

---

## Engineering Challenges Solved

### 1. Zero-Storage Audio Synthesis
Traditional audio apps serve static files. Mind-Sync serves **mathematical specifications**. The backend never touches an audio file. The frontend synthesizes sound directly from frequency parameters in real-time using the browser's PCM-level audio graph.

### 2. Deterministic DSP State
The DSP engine is a **pure function**: identical input parameters always produce identical output blueprints. This is enforced via SHA-256 checksums of synthesis parameters. Any caching layer can validate blueprint identity without re-computation.

### 3. Glitch-Free Live Parameter Updates
When a user moves a slider during playback, the system must update the audio **without clicks, pops, or interruption**. This is solved via Web Audio API `linearRampToValueAtTime` parameter automation вҖ” frequency and gain transitions are scheduled on the audio thread, not the UI thread.

### 4. Psychoacoustic Frequency Mapping
The three user sliders (Focus Depth, Calm Level, Energy Level) map to brainwave frequency bands via a weighted centroid algorithm derived from peer-reviewed neuroscience literature. This is not arbitrary linear mapping вҖ” it correctly reflects the psychoacoustic properties of each brainwave state.

### 5. Pink Noise via Voss-McCartney Algorithm
True 1/f pink noise (the most natural-sounding noise profile) cannot be generated by simple random sampling. Mind-Sync implements the **Voss-McCartney algorithm** in Dart, generating a statistically correct 1/f power spectral density directly in the audio buffer.

## Design Decisions & Tradeoffs

### Why Go for the Backend?
Go's compile-time type safety, zero-dependency binary output, and first-class HTTP tooling made it ideal for a high-performance, stateless computation server. The DSP engine is a pure function вҖ” no goroutine state, no shared memory вҖ” making it trivially safe to run under any concurrency model.

### Why BLoC for Flutter State Management?
BLoC enforces strict separation between UI and business logic. The `AudioEngineBloc` is a formal state machine with sealed event and state hierarchies. This means every state transition is explicit, traceable, and testable вҖ” critical for an audio engine where state bugs cause audible glitches.

### Why SQLite over Postgres/Redis?
The constraint was "runs on `git clone`". SQLite in WAL mode provides ACID transactions, concurrent reads, and durability without any external process. For this use case (single-user, moderate write volume), WAL SQLite outperforms the operational burden of a separate database server.

### Why Not serve Audio from the Backend?
Generating real-time audio streams server-side would require: a streaming protocol (WebSockets/SSE), real-time buffer generation (PCM at 44100 Hz), and significant compute per connected client. The browser's Web Audio API already has a hardware-accelerated, sample-accurate synthesis engine. The backend's role is to *compute the parameters*; the frontend's role is to *synthesize the sound*. This separation enables instant sub-50ms parameter updates without network round-trips during playback.

## Architecture Principles Applied

| Principle                        | Implementation |
|----------------------------------|----------------|
| **Separation of Concerns**       | Domain, Application, Infrastructure, Interface layers never import each other out of order |
| **Dependency Inversion**         | `NeuralSignatureRepository` is a Go interface; SQLite is one implementation |
| **Immutability**                 | All BLoC states, all Flutter models are `final` / `const` / `Equatable` |
| **Deterministic DSP**            | Same params вҶ’ same blueprint вҶ’ same checksum. Always. |
| **Repository Pattern**           | Persistence is behind a strict interface; the use cases never see SQL |
| **Anti-Corruption Layer**        | DTO вҶ” Entity mapping happens only in use cases; domain never knows about JSON |
| **Graceful Degradation**         | Session tracking failure is non-fatal; audio continues uninterrupted |
